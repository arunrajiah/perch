# F-Droid Submission Notes

This document captures gotchas discovered during the F-Droid metadata research. Read before submitting.

## Recommended path: IzzyOnDroid first

[IzzyOnDroid](https://apt.izzysoft.de/fdroid/) takes pre-built APKs directly from GitHub Releases — no source build required. It is included in many F-Droid clients as an extra repository. Submit there first, prove the app works in the ecosystem, then tackle the full fdroiddata MR.

**IzzyOnDroid submission:** https://gitlab.com/IzzyOnDroid/repo/-/issues (open a new issue with the GitHub Releases URL)

## Major blockers for full fdroiddata

### 1. `android/` is not committed to the repo (biggest issue)

BirdEcho uses Expo Managed Workflow — `android/` is gitignored and generated by `expo prebuild`. F-Droid's build server must run `expo prebuild` before Gradle, which is non-standard and will likely get pushback from reviewers.

**Recommended fix:** commit the generated `android/` directory on a dedicated `fdroid` branch or add it to `main`. Point the recipe's `commit:` at the commit that includes it. Keep using EAS for your own release builds; maintain the `android/` snapshot separately for F-Droid.

### 2. Reproducible builds

Expo + Hermes is not reproducible today (timestamps in prebuild output, non-deterministic Hermes bytecode, Metro bundle hashes). The `Binaries:` + `AllowedAPKSigningKeys` path will not work without significant effort.

This means F-Droid will sign the APK with their key — users who installed from GitHub Releases **cannot receive F-Droid updates without reinstalling**. This is acceptable for the initial submission.

### 3. Sentry anti-feature

`@sentry/react-native`'s Java/Kotlin layer compiles into the APK regardless of whether the DSN env var is set. F-Droid's scanner will detect it and require the `Tracking` anti-feature.

Options:
- **Keep it**: declare `Tracking`. App is hidden by default in F-Droid client (users must enable anti-feature display). ~98 apps carry this tag — it's manageable.
- **Strip it for F-Droid builds**: add `sed -i '/"@sentry\/react-native"/d' package.json` to the `init:` block. Produces a Sentry-free binary. Remove the `AntiFeatures` entry. Cleaner.

The YAML recipe includes a commented-out line for the strip approach.

## Checklist before opening the fdroiddata MR

- [ ] Replace `commit: HEAD` with the exact commit SHA of the release tag
- [ ] Update the Node.js tarball URL and SHA-256 hash in `sudo:`
- [ ] Either commit `android/` to the repo (recommended) or confirm the prebuild approach works in an `fdroid build` local test
- [ ] Decide on Sentry: keep `Tracking` or uncomment the strip line
- [ ] Uncomment `AllowedAPKSigningKeys` + `Binaries:` if reproducible builds are ever achieved
- [ ] Test locally: `fdroid build dev.arunrajiah.birdecho` (requires fdroidserver)

## F-Droid process overview

1. Fork https://gitlab.com/fdroid/fdroiddata
2. Create `metadata/dev.arunrajiah.birdecho.yml` from the draft in this directory
3. Run `fdroid lint metadata/dev.arunrajiah.birdecho.yml` locally
4. Run `fdroid build --on-server dev.arunrajiah.birdecho` locally
5. Open a Merge Request; F-Droid bots and a human reviewer will check it
6. Review typically takes 2–8 weeks
